## 1.5. 摘要
多模态大语言模型在视觉描述和视觉问答等理解任务上取得了显著成功。然而，由于输出格式和优化目标的差异，它们在基于嵌入的检索任务上的直接应用仍然具有挑战性。先前的方法通常采用对比微调来使 MLLMs 适应检索任务，但代价是失去了其生成能力。本文认为，生成和嵌入任务从根本上依赖于共享的认知机制，特别是跨模态表示对齐和上下文理解。为此，我们提出了 <strong>CREM (Compression-driven Representation Enhanced Model，压缩驱动表示增强模型)</strong>，这是一个统一的框架，在增强多模态表示以用于检索的同时保留了生成能力。具体而言，我们引入了一种基于压缩的提示设计，其中包含可学习的 **合唱词元** 来聚合多模态语义，以及一种压缩驱动的训练策略，通过压缩感知注意力整合对比和生成目标。大量实验表明，CREM 在 MMEB 上实现了最先进的检索性能，同时在多个理解基准上保持了强大的生成性能。我们的发现表明，在提出的压缩驱动范式下，生成监督可以进一步提高 MLLMs 的表示质量。

# 2. 整体概括

## 2.1. 研究背景与动机
**核心问题：**
多模态大语言模型在生成任务（如描述图片、回答问题）上表现出色，但在检索任务（如找图、搜文）上存在短板。现有的解决方案通常通过对比学习将 MLLMs 微调为嵌入模型，但这会导致模型丧失原有的生成能力（即“灾难性遗忘”）。如何在增强检索能力的同时，不损失生成能力，是该领域面临的关键挑战。

**重要性与挑战：**
在实际应用中，一个理想的系统既能进行精准的检索，又能进行流畅的生成和对话。维护两套独立的模型（一个检索模型，一个生成模型）会带来巨大的计算资源开销和系统复杂度。现有研究（如 CAFe）尝试统一这两个目标，但往往只是简单地将两个任务的损失函数相加，未能从机制上解决生成与检索之间的表示冲突。

**切入点与创新思路：**
本文的切入点在于<strong>“压缩”</strong>。作者观察到，检索任务本质上需要将高维的视觉和文本信息压缩为一个紧凑的向量表示，而生成任务虽然需要处理详细的 token 序列，但也依赖于对核心语义的捕捉。CREM 提出引入一组特殊的、可学习的 <strong>“合唱词元”</strong>，强制模型将多模态信息压缩到这些 token 中。通过设计特殊的注意力掩码，让生成任务必须基于这些压缩后的 token 进行，从而在训练过程中统一了检索（用压缩 token 做嵌入）和生成（用压缩 token 做上下文）的优化目标。

## 2.2. 核心贡献/主要发现
**主要贡献：**
1.  **压缩感知提示设计：** 提出了引入可学习合唱词元的提示结构，作为连接嵌入和生成任务的桥梁，使得检索嵌入和生成词元共享一致的表示空间。
2.  **压缩驱动训练策略：** 开发了一种联合优化对比学习和语言建模的统一框架。通过压缩感知注意力机制和生成数据混合策略，实现了两种范式间的动态跨任务交互和高效知识共享。
3.  **性能验证：** 在 MMEB 检索基准上取得了最先进的性能，同时在 MMB、MMMU 等理解基准上保持了强大的生成能力。证明了生成监督可以有效提升检索性能。

**关键发现：**
- 生成任务和检索任务并非互斥，通过统一的压缩表示，两者可以相互促进。
- 即使将视觉 token 压缩 80 倍（仅保留合唱词元），模型仍能保留约 83% 的生成响应质量，说明压缩 token 保留了足够的信息。

# 3. 预备知识与相关工作

## 3.1. 基础概念
为了理解本文，读者需要掌握以下核心概念：

- **多模态大语言模型:** 这类模型（如 LLaVA, Qwen-VL）不仅处理文本，还能处理图像。通常通过一个视觉编码器将图像转换为视觉特征，然后通过投影层将其映射到 LLM 的词元空间，使 LLM 能够像处理文本一样“看懂”图片。
- **检索与嵌入:** 在检索任务中，模型的目标不是生成下一个词，而是将输入（图像或文本）映射为一个固定长度的向量。通过计算向量之间的相似度（如余弦相似度）来判断匹配程度。
- **对比学习:** 训练嵌入模型的主流方法。核心思想是拉近正样本对（如“一张猫的图”和“一只猫”）的距离，推远负样本对（如“一张猫的图”和“一只狗”）的距离。常用的损失函数是 InfoNCE Loss。
- **注意力机制:** Transformer 模型的核心，允许模型在处理某个词时关注输入序列中的其他词。本文使用了特殊的“注意力掩码”来控制信息的流动方向。

## 3.2. 前人工作
作者在文中提到了几类重要的相关工作：

1.  **多模态表示学习:** 如 CLIP, CoCa 等模型，通过对比学习学习图像和文本的对齐表示。近期的工作如 VLM2Vec, E5-V 尝试利用 MLLMs 的强大能力来构建更好的嵌入模型，但往往牺牲了生成能力。
2.  **统一生成嵌入模型:** 如 CAFe, GRITLM, MM-GEM。这些工作试图在一个模型中同时支持生成和检索。CAFe 通过简单的损失相加来联合优化，但本文认为这种方法未能充分挖掘两个任务之间的内在联系。
3.  **多模态词元压缩:** 为了解决 MLLMs 处理高分辨率图像时 token 数量过多导致的计算量爆炸问题，前人提出了各种压缩方法，如 Perceiver Resampler, Q-Former 等。本文利用了这一思想，但将其用于统一检索和生成任务。

**补充核心公式：InfoNCE Loss**
虽然本文主要关注方法创新，但其检索部分依赖于对比学习。标准的 InfoNCE Loss 公式如下：
$$
\mathcal{L}_{\text{InfoNCE}} = -\log \frac{\exp(\text{sim}(z_i, z_j) / \tau)}{\sum_{k=1}^{K} \exp(\text{sim}(z_i, z_k) / \tau)}
$$
其中 $z_i$ 和 $z_j$ 是正样本对的表示，$\text{sim}$ 是相似度函数，$\tau$ 是温度系数。该公式的目的是最大化正样本对的相似度相对于所有样本（包括负样本）相似度的比例。

## 3.3. 技术演进
该领域经历了从“双塔模型”（CLIP）到“基于 MLLM 的嵌入模型”（VLM2Vec），再到“统一生成与嵌入模型”（CAFe, CREM）的演进过程。早期的双塔模型生成能力弱；基于 MLLM 的嵌入模型虽然利用了强大的预训练知识，但丢失了生成能力；最新的趋势是试图在单一架构中同时保留两种能力。CREM 处于这一演进的最前沿，通过引入压缩机制解决了前人工作中任务割裂的问题。

## 3.4. 差异化分析
本文方法与 CAFe 等前人工作的核心区别在于：
- **CAFe:** 使用不同的提示分别引导生成和嵌入任务，并简单地将两个损失相加。这实际上将两个任务视为独立的，忽略了它们在底层表示上的潜在冲突。
- **CREM:** 引入了**合唱词元**作为统一的表示载体。无论是做检索还是做生成，模型都必须将信息压缩到这些 token 中。通过**压缩感知注意力掩码**，强制生成任务只能看到压缩后的信息，从而在机制上保证了两个任务优化目标的一致性，而非简单的损失叠加。

# 4. 方法论

## 4.1. 方法原理
CREM 的核心思想是<strong>“压缩驱动”</strong>。作者认为，检索任务本质上就是寻找一个高质量的压缩表示，而生成任务虽然输出长序列，但其核心语义也可以被压缩。因此，CREM 在输入序列中插入一组可学习的特殊 token（称为**合唱词元**），并通过修改注意力掩码，强制模型将原始的视觉和文本信息“压缩”进这些合唱词元中。在训练时，检索任务使用这些合唱词元的输出作为嵌入向量，生成任务则基于这些合唱词元来预测答案。这样，两个任务就在同一个压缩表示空间上进行了联合优化。

## 4.2. 核心方法详解 (逐层深入)

### 4.2.1. 压缩感知提示设计
首先，模型需要构建一个统一的输入格式。传统的生成模型使用类似 $<image> Question: ... Answer: ...$ 的格式，而嵌入模型通常使用特殊的指令。CREM 将两者统一，引入了合唱词元 $\mathcal{U}$。

**提示模板构建：**
具体的提示设计如下：
`System: You are a helpful assistant that can represent and understand multimodal inputs.`
$User: <image> [eInst] <chorus> [gInst]$
$Assistant: <answer>$

其中：
- $<image>$ 是视觉词元序列 $\mathcal{V}$。
- `[eInst]` 是嵌入指令（如 "Represent the image."）。
- $<chorus>$ 是插入的合唱词元序列 $\mathcal{U}$。
- `[gInst]` 是生成指令（如 "Describe the image."）。
- $<answer>$ 是生成的回答序列词元 $\mathcal{A}$。

**信息压缩与互信息最大化：**
模型的目标是让合唱词元 $\mathcal{U}$ 最大化与原始多模态输入（视觉 $\mathcal{V}$ 和文本 $\mathcal{T}$）之间的互信息。这可以用以下公式表示：

$$
\mathbb { I } _ { \mathcal { V } , \mathcal { T } ; \mathcal { U } } = D _ { \mathrm { K L } } \left( p( \mathcal { V } , \mathcal { T } , \mathcal { U } ) \parallel p ( \mathcal { V } , \mathcal { T } ) \otimes p ( \mathcal { U } ) \right)
$$

**公式解释：**
- $\mathbb{I}_{\mathcal{V}, \mathcal{T}; \mathcal{U}}$: 表示视觉输入 $\mathcal{V}$、文本输入 $\mathcal{T}$ 与合唱词元 $\mathcal{U}$ 之间的互信息。互信息越高，说明 $\mathcal{U}$ 包含了越多的关于 $\mathcal{V}$ 和 $\mathcal{T}$ 的信息。
- $D_{\mathrm{KL}}$: KL 散度，用于衡量两个概率分布之间的差异。
- $p(\mathcal{V}, \mathcal{T}, \mathcal{U})$: 联合概率分布。
- $p(\mathcal{V}, \mathcal{T}) \otimes p(\mathcal{U})$: 边缘概率分布的乘积。
- 该公式通过最大化 KL 散度，实际上是在迫使合唱词元 $\mathcal{U}$ 尽可能多地捕捉原始输入 $\mathcal{V}$ 和 $\mathcal{T}$ 中的信息，即实现信息的有效压缩。

### 4.2.2. 压缩感知注意力机制
为了实现上述压缩目标，CREM 设计了一个特殊的注意力掩码 $M$，严格控制信息的流动方向。

**注意力掩码定义：**
标准的因果注意力掩码通常只允许当前词关注之前的词。CREM 对其进行了修改，定义如下：

$$
M _ { i j } = { \left\{ \begin{array} { l l } { 0 , } & { { \mathrm { i ~ f ~ } } i \in ( \mathscr { Q } , \mathscr { A } ) { \mathrm { ~ a n d ~ } } j \in ( \mathscr { V } , \mathscr { T } ) , } \\ { 1 ( j \leq i ) , } & { { \mathrm { o t h e r w i s e } } . \end{array} \right. }
$$

**公式解释：**
- $M_{ij}$: 表示第 $i$ 个位置的词是否可以关注第 $j$ 个位置的词。1 表示允许关注（可见），0 表示禁止关注（被 Mask 掉）。
- $i \in (\mathscr{Q}, \mathscr{A})$: $i$ 是问题词元或回答词元。
- $j \in (\mathscr{V}, \mathscr{T})$: $j$ 是原始视觉词元或文本词元。
- **核心逻辑：** 如果当前位置 $i$ 是问题或回答部分，且被关注的位置 $j$ 是原始的视觉或文本输入，则 $M_{ij}=0$，即**禁止关注**。这意味着生成部分（问题和回答）**只能看到**合唱词元 $\mathcal{U}$，而无法直接看到原始的 $\mathcal{V}$ 和 $\mathcal{T}$。这强制模型必须将所有必要信息先压缩到 $\mathcal{U}$ 中，否则无法完成生成任务。
- $1 (j \leq i)$: 在其他情况下（即合唱词元关注原始输入，或者生成部分关注合唱词元），保持标准的因果注意力机制。

### 4.2.3. 压缩驱动训练策略
在构建了上述结构后，CREM 采用联合训练策略，同时优化检索和生成目标。

**1. 数据混合策略：**
为了增强模型的一致性和泛化能力，CREM 混合了两种数据：
- **同构数据混合:** 检索样本和生成样本来自同一数据源。例如，对于一张图，既有检索用的对，也有基于该图生成的 QA 数据。这有助于增强跨任务的一致性。
- **异构数据混合:** 检索样本和生成样本来自不同数据源。检索用 MMEB，生成用 ShareGPT-4V。这有助于保持模型在开放域生成上的泛化能力。

**2. 检索损失:**
对于检索任务，使用标准的对比学习损失。基于图像-文本对数据，构建正样本对，并计算 InfoNCE 损失：

$$
\mathcal { L } _ { \mathrm { r } } = - \log \frac { \phi ( \mathbf { h } _ { q } , \mathbf { h } _ { t ^ { + } } ) } { \phi ( \mathbf { h } _ { q } , \mathbf { h } _ { t ^ { + } } ) + \displaystyle \sum _ { t ^ { - } \in \mathbb { N } } \phi ( \mathbf { h } _ { q } , \mathbf { h } _ { t ^ { - } } ) }
$$

**公式解释：**
- $\mathcal{L}_{\mathrm{r}}$: 检索任务的损失函数。
- $\mathbf{h}_q, \mathbf{h}_t$: 分别为查询和目标的表示向量。这些向量是通过对最后一层的 $k$ 个合唱词元进行平均池化得到的。
- $t^+$: 正样本目标。
- $\mathbb{N}$: 负样本集合。
- $\phi$: 相似度计算函数，文中使用的是带温度系数的余弦相似度：$\phi(\mathbf{h}_q, \mathbf{h}_t) = \exp(\cos(\mathbf{h}_q, \mathbf{h}_t) / \tau)$。
- 该公式的目标是最大化正样本对的相似度分数在所有样本（正样本+负样本）相似度分数总和中的占比。

**3. 生成损失:**
为了在保持生成能力的同时训练压缩能力，CREM 引入了**随机压缩驱动语言建模损失**。

$$
\mathcal { L } _ { \mathrm { g } } = - \frac { 1 } { T } \sum _ { t = 1 } ^ { T } \log p _ { C } ( y _ { t } \mid y _ { < t } , \mathcal { U } , 1 _ { z = 0 } ( \mathcal { V } , \mathcal { T } ) ) .
$$

**公式解释：**
- $\mathcal{L}_{\mathrm{g}}$: 生成任务的损失函数（标准的交叉熵损失）。
- $p_C$: 模型预测下一个词的概率。
- $y_t$: 第 $t$ 个真实的词元。
- $y_{<t}$: 之前已生成的词元。
- $\mathcal{U}$: 合唱词元。
- $1_{z=0}(\mathcal{V}, \mathcal{T})$: 这是一个指示函数项。$z$ 是一个伯努利随机变量 $z \sim \text{Bernoulli}(p)$。
  - 当 $z=0$ 时（概率为 `1-p`），该项为 $(\mathcal{V}, \mathcal{T})$，意味着模型可以看到完整的原始输入（标准生成模式）。
  - 当 $z=1$ 时（概率为 $p$），该项为空（或被 Mask），意味着模型只能看到合唱词元 $\mathcal{U}$（压缩生成模式）。
- 这种随机机制强迫模型在学习过程中，既要适应基于压缩信息的生成，又要保持基于原始信息的生成能力，从而实现鲁棒性。

**4. 总损失:**
最终的总损失是检索损失和生成损失的加权和：

$$
{ \mathcal { L } } = \alpha _ { \mathrm { { r } } } { \mathcal { L } } _ { \mathrm { r } } + \alpha _ { \mathrm { { g } } } { \mathcal { L } } _ { \mathrm { g } }
$$

其中 $\alpha_{\mathrm{r}}$ 和 $\alpha_{\mathrm{g}}$ 是权重超参数。

### 4.2.4. 多任务推理模式
训练完成后，CREM 支持三种推理模式，如下图（原文 Figure 3）所示：

![Figure 3. CREM Inference Modes. (a) Retrieval embeddings are derived from pooled chorus tokens. (b) Native next-token prediction is performed with full access to all vision tokens (Nat.). (c) Efficient inference is achieved by pruning vision tokens and reducing KV caches (Comp.).](images/3.jpg)
*该图像是示意图，展示了 CREM 模型在多模态检索和生成任务中的推理模式。图(a)展示了多模态检索中从聚合的合唱令牌得到的嵌入，图(b)显示了在多模态语言生成中对所有视觉令牌的完整访问，图(c)描述了基于压缩的多模态语言生成过程，其中包括预填充和解码阶段。*

1.  **检索模式:** 只保留合唱词元，丢弃原始视觉和文本 token。对合唱词元进行平均池化得到嵌入向量。这极大地减少了计算量。
2.  **原生生成模式:** 不插入合唱词元，直接使用原始的完整输入进行推理。这保证了模型在需要精细细节时的生成质量。
3.  **压缩生成模式:** 插入合唱词元，利用它们压缩原始信息，并在解码时丢弃原始 KV Cache，仅保留合唱词元的 KV Cache。这实现了高效的长上下文推理。

# 5. 实验设置

## 5.1. 数据集
实验使用了以下主要数据集：

1.  **MMEB (Massive Multimodal Embedding Benchmark):**
    -   **来源与规模:** 这是一个大规模的多模态嵌入基准测试，包含 36 个数据集，分为 4 个元任务类别：分类、视觉问答、检索和视觉定位。其中 20 个作为分布内（IND）数据，16 个作为分布外（OOD）数据。
    -   **特点:** 涵盖了广泛的多模态任务，能够全面评估模型的检索和泛化能力。
    -   **用途:** 用于训练和评估检索性能。

2.  **ShareGPT-4V:**
    -   **来源:** 一个高质量的图像-文本对话数据集。
    -   **特点:** 包含丰富的视觉描述和复杂的对话。
    -   **用途:** 作为异构生成数据，用于增强模型的 token 压缩能力和泛化能力。

3.  **合成 QA 数据:**
    -   **来源:** 使用 Qwen2.5-VL-7B 模型基于 MMEB 中的样本合成。
    -   **用途:** 作为同构生成数据，确保检索和生成任务在数据层面的一致性。

## 5.2. 评估指标

### 5.2.1. Precision@1 (P@1)
1.  **概念定义:** 在检索任务中，Precision@1 衡量的是模型返回的第一个结果（即最相似的结果）是否是正确的（即是否为 Ground Truth）。它关注的是“排在第一位”的准确率，是检索质量最直观的体现。
2.  **数学公式:**
    $$ \text{Precision@1} = \frac{1}{N} \sum_{i=1}^{N} \mathbb{I}(\text{rank}(q_i, t_i^+) = 1) $$
3.  **符号解释:**
    -   $N$: 查询的总数。
    -   $q_i$: 第 $i$ 个查询。
    -   $t_i^+$: 第 $i$ 个查询对应的正确目标。
    -   $\text{rank}(q_i, t_i^+)$: 正确目标 $t_i^+$ 在查询 $q_i$ 的检索结果列表中的排名。
    -   $\mathbb{I}(\cdot)$: 指示函数，如果条件为真则取 1，否则取 0。

### 5.2.2. 生成任务通用指标
对于生成任务（如 MMBench, MMVet 等），论文主要报告各基准测试的官方评分（通常是准确率 Accuracy 或综合得分），这些指标衡量模型生成答案的正确性和质量。

## 5.3. 对比基线
论文将 CREM 与以下几类基线模型进行了比较：

-   **传统双塔模型:** CLIP, OpenCLIP。作为早期的弱基线。
-   **MLLM-based 嵌入模型:** VLM2Vec, VLM2Vec-V2, E5-V, mmE5, UNITE, LLaVE。这些是当前检索任务的主流强基线，但通常缺乏生成能力。
-   **统一生成嵌入模型:** CAFe。这是最直接的相关工作，试图统一生成和检索。
-   **生成基线:** Qwen2-VL (原始模型), CREM_G (仅生成微调), CREM_R (仅检索微调)。用于消融分析。

# 7. 总结与思考

## 7.1. 结论总结
CREM 提出了一个创新的压缩驱动框架，成功解决了多模态大语言模型在检索与生成任务之间的权衡问题。通过引入可学习的合唱词元和压缩感知注意力机制，CREM 强制模型将多模态信息压缩到一个统一的表示空间中。实验证明，该方法不仅显著提升了检索性能（达到 SOTA），还有效保留了模型的生成能力。此外，该压缩机制还为降低推理时的 KV Cache 开销提供了一条有效路径。

## 7.2. 局限性与未来工作
作者在论文中指出了以下局限性：
1.  **固定 Token 数量:** 目前合唱词元的数量是固定的，而不同任务可能需要不同大小的压缩表示。未来可以探索动态或任务感知的 token 分配机制。
2.  **OCR 性能下降:** 基于压缩的推理在 OCR 任务上表现较差，可能是因为丢失了细粒度的视觉细节。未来可以通过加入更多 OCR 特定的数据来缓解这一问题。
3.  **数据依赖:** 目前的训练主要依赖 MMEB 和 ShareGPT-4V，引入更多样化、高质量的数据可能会进一步提升泛化能力。

## 7.3. 个人启发与批判
**启发：**
-   **机制设计优于损失叠加:** CREM 的成功表明，解决多任务冲突的关键往往不在于如何组合损失函数，而在于设计合理的模型架构（如注意力掩码）来引导信息流。这种“机制优先”的思路值得借鉴。
-   **压缩即理解:** 压缩不仅仅是加速的手段，更是理解的一种形式。如果模型能将长序列压缩成短序列而不丢失语义，说明它真正“理解”了内容。这为未来的模型设计提供了新的视角。

**批判与思考：**
-   **掩码的刚性:** 虽然注意力掩码有效地强制了信息压缩，但也可能限制了模型在某些需要精细细节的任务上的表现（如文中提到的 OCR）。是否存在一种更柔性的机制，既能鼓励压缩，又允许在必要时“回溯”原始细节？
-   **计算开销:** 虽然推理时 KV Cache 减少了，但在训练阶段，由于需要同时计算检索和生成损失，且涉及数据混合，训练的复杂度和资源需求可能较高。该方法在小规模资源受限场景下的可复现性值得观察。
-   **评估的全面性:** 虽然在 MMEB 上表现优异，但在一些特定的长尾任务或需要极高空间精度的任务（如精确的像素级操作）上，压缩表示可能存在天然的上限。未来的工作可以更深入地探讨压缩率与任务能力边界的关系。
