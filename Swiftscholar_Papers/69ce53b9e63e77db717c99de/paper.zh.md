# HippoCamp：在个人计算机上对上下文智能体进行基准测试

我们提出了HippoCamp，这是一个旨在评估智能体在多模态文件管理能力的新基准。与现有的聚焦于网页交互、工具使用或软件自动化等通用设置的智能体基准不同，HippoCamp在以用户为中心的环境中评估智能体，以建模个体用户档案并从大量个人文件中进行上下文感知推理。我们的基准在真实世界的个人档案上实例化了设备级文件系统，涵盖了跨越多种模态的42.4 GB数据，涉及超过2000个真实文件。在原始文件的基础上，我们构建了581对问答，以评估智能体在搜索、证据感知和多步骤推理方面的能力。为了促进细致的分析，我们提供了46.1K个密集标注的结构化轨迹，以进行逐步失败诊断。我们在HippoCamp上评估了各种最先进的多模态大语言模型（MLLM）和智能方法。我们的全面实验揭示了显著的性能差距：即使是最先进的商业模型在用户建模方面的准确率也仅为$48.3\%$，在长时间检索和密集个人文件系统中的跨模态推理方面尤其挣扎。此外，我们的逐步失败诊断确定了多模态感知和证据基础作为主要瓶颈。最终，HippoCamp揭示了当前智能体在现实、以用户为中心的环境中的关键局限性，并为开发下一代个人AI助手提供了坚实的基础。关键词：多模态智能体；文件系统；上下文基准评估；个性化记忆。 日期：2026年4月1日 项目页面：https://hippocamp-ai.github.io 数据可视化：https://hippocamp-ai.github.io/hippocamp/

大型多模态模型（LMMs）和自主系统的兴起加速了朝智能助手的进展，这些助手能够在感知、推理和行动之间进行操作（Xi等，2023；Sumers等，2024；Yao等，2025）。此类系统的一个自然和重要的应用是个性化数字环境（Wang等，2024b），在这些环境中，智能体管理、检索和推理用户的异构数字数据集（Lietal 2024b）。与基于网络或任务特定的智能体（We等，205；Mialon等，2；Yao等，2022）不同，操作于本地数字环境中的个性化智能体必须在不断变化的设备中处理多模态数据文件、各种格式以及长期行为轨迹（Viana，2019；Wu等，2024）。在这种上下文中进行推理需要整合跨模态的感知与过去交互的检索和反思，与人类海马体在上下文记忆巩固和回忆中的作用相似。

然而，尽管在长时序推理和多模态检索方面取得了近期进展（Liu 等，2025），仍然不存在评估智能体理解、回忆和推理巨量个性化多模态文件系统能力的标准化基准。虽然近期的研究关注于文档级多模态检索（Dong 等，2025a）或个性化工具使用规划（Xiu 等，2026），但没有一个针对个人计算环境的规模和异质性。现有评估主要针对通用领域，比如网络自动化（Gure 等，2024；Song 等，2024；Koh 等，2024）、代码生成（Yu 等，2024）、文档理解（Ouyang 等，2025）或具体任务规划（Sadhu 等，2025），而忽视了个性化文件系统中蕴含的丰富信息。尽管这些基准提高了感知和工具使用能力，但它们在与用户个人上下文脱节的孤立、目标特定的场景中进行，无法捕获长期持续性、复杂的交叉引用以及随时间推移密切相关的多模态文件的验证，并且未能提供符合实际个人计算需求的个性化推理。为填补这一空白，我们引入了HippoCamp，这是一个用于评估在真实个人计算环境中记忆增强智能体的基准，如图1所示。HippoCamp构建了三个基于真实用户数据的代表性个人计算环境。每个环境不是字面意义上的三分法分类，而是一个高维特征空间的典型实例。它们共同揭示了设备规模的文件系统挑战，如深层次的层次组织、多样化的长尾文件格式和跨模态证据依赖。该基准涵盖了两类代表性任务，反映了真实计算机使用的场景：事实保留和用户画像。每个任务要求智能体结合搜索、感知和推理，系统地评估个性化多模态智能。总之，HippoCamp 有三个主要贡献：1. 真实的个人计算环境。我们通过构建三个不同的、文件密集型的个人资料，忠实地模拟用户级数字生态系统。这个设计捕捉了长期的连续性、独特的文件夹结构以及现实世界个人文件系统的相互关联性。2. 设备规模的语料库，配备密集监督。我们提供了一个庞大的数据集，包含超过2000个异质文件（总计42.4 GB），并附有581个以用户需求为驱动、以证据为基础的查询。为了支持对不同深度和视角下的智能体能力进行严格评估，我们在多个细度层次上包含了46.1K的细粒度标注。3. 全面的智能体能力评估。我们制定了一个严格的问题回答基准，基于两个核心任务类别：事实保留：检索特定信息，以及用户画像：推断用户偏好。这些任务超越了简单的检索，要求智能体执行多步骤行为，能够无缝整合文件系统搜索、多模态证据感知和跨相关文件的个性化推理。

多模态上下文智能体的基准测试。上下文检索基准测试涵盖了从文本中心数据集（Yang et al., 2018; Petroni et al., 2021; Tang and Yang, 2024; Friel et al., 2025）到基于更丰富证据需求的多模态和智能体设置。MultimodalQA (Talmor et al., 2021) 支持针对文本、图像和表格的跨模态基础，而 WebQA (Chang and Bisk, 2022) 研究从包含干扰项的候选池中进行检索。文档基准测试如 M3DocRAG (Cho et al., 2024) 和 MMDocRAG (Dong et al., 2025b) 强调在噪声下的细粒度选择。智能体基准测试则从静态语料库转向互动环境：WebShop (Yao et al., 2022) 和 PaperBench (Starace et al., 2025) 评估目标驱动的行动序列，而 MetaTool (Huang et al., 2024)、MINT (Wang et al., 2024a)、InfeepSeek (Xi et al., 2025) 和 BrowseComp (Wei et al., 2025) 则探讨工具使用、多轮推理以及网络探索。然而，这些基准测试大多假设公共数据和完全可观察的状态，因此并未评估长期存在的个性化上下文或分布在用户设备上的异构多模态证据。真实的个人环境包含身份线索、行为历史和跨多种文件类型的纵向记录——这些信号在当前评估中大多缺失。

具有记忆和个性化的智能系统。智能系统在交互环境中执行多步骤工作流程，因此需要上下文整合；记忆对于长时间一致性至关重要。先前的研究探讨了（i）基于轨迹的记忆，例如，通过微调或强化学习内化经验（张等，2024；傅等，2025）；（ii）基于检索的记忆，例如，存储和检索过去的事件或工具轨迹（赵等，2024；罗等，2025）；以及（iii）技能蒸馏，将可重用技能压缩为推理时模块（王等，2025；郑等，2025）。另一方面，还有研究构建结构化的情节/语义记忆，以组织交互历史并提高多步骤的可靠性（张等，2025a；杨等，2025）。基于这些机制，个性化在用户层面上实现记忆：PersonaAgent（张等，2025b）维护用户特定的情节/语义记录，而最近的系统学习用户知识图谱、偏好嵌入或长期历史，以在无需参数更新的情况下进行适应（王等，2024b；李等，2025）。此外，Telemem（陈等，2026）将以用户为基础的交互整合成叙事和多模态情节记忆，从而实现个性化的对话和视觉体验检索。然而，现有的评估规模较小或为合成，通常仅限于狭窄的模态（例如，文本/网页），未能反映个人计算中异质且不断发展的上下文，其中证据跨越所有五种模态，促使相应规模和多样性基准的建立，以匹配用户文件生态系统。

多模态上下文检索。检索增强生成（RAG）利用检索作为外部记忆，以获取超出上下文窗口的证据，而以推理为中心的模型如DeepSeek-R1（Guo等，2025a）在多步骤推理过程中细化中间痕迹并调用工具。多模态RAG将文本检索推广到异构证据：文档系统（Cho等，2024；Dong等，2025a）编码布局和混合文本图像内容以实现细粒度的基础；图文框架（Zhan等，2025；Gu等，2025b）融合视觉和文本特征以实现跨模态对齐；视频方法如VideoRAG（Ren等，2025）索引时间片段以实现远程视觉检索。尽管覆盖面更广，top $k$ 检索在提示分散于多个文件或长时间跨度时仍然脆弱。推理模型通过生成中间痕迹来协调检索部分缓解这一问题：Search-R1（Jin等，2025）和MMSearch-R1（Wu等，2025）迭代地细化查询，Ego-R1（Tian等，2025）将这一范式扩展到自我中心流。 然而，RAG和推理模型主要假设公共或任务限定的检索空间，而不是现实用户生态系统中的个性化、长期的多模态上下文。我们的基准直接针对这一设置（见表1），评估相关信号跨所有模态的累积情况，在真实的、用户特定的数字环境中，检索和推理都必须在其中进行操作。

HippoCamp 是一个用于评估记忆增强智能体在现实设备驻留个人文件系统中的基准。它将个人计算建模为一个多模态、长尾的信息空间，其中日常文物和信息交互在这一空间中变得动态多样。我们将个性化文件理解形式化为开放式、基于证据的问题回答。图 2 并非仅仅提供数据集概述，而是详细说明了基准的监督层级：它将从低级、局部证据和行动轨迹的注释组织到结构化轨迹和任务级评估，随着抽象程度的提高，最终到达用户级记忆。这一层级明确了 HippoCamp 如何在多个粒度层面支持诊断，从智能体是否能够在单个文件中定位证据，到它是否能够将基于证据的中间步骤组合成连贯的答案，涵盖较长时间的上下文。在这一结构中，我们定义了两种任务类型，随着聚合需求的增加而增强。事实保留要求检索和推理关键信息，而概括位于层级的顶部，要求智能体在时间上综合多个基于证据的事实，形成连贯的用户级推论，如偏好、行为模式、调度信息、回顾性反思和工作流程。解决这两项任务依赖于三种相互关联的能力：搜索以在大型异构系统中定位相关文件、感知以解释多模态内容，以及推理以将跨文件、跨时间的证据整合成准确的、上下文感知的答案。

# 源。

HippoCamp 是通过对 100 余名参与者进行访谈而得出的，这些参与者的选择旨在反映一般个人计算设置的特征，参与者的系统表现出稳定的行为规律和完整的长周期个人数据痕迹，这些痕迹支持可审核的用户级推理跨层级参考。然后，我们将选定参与者的数据汇总成一致的原型档案，通过匹配类型/模态分布和高层次的组织模式，同时涵盖多样的人口统计、社会经济、职业和生活方式维度。经过汇总，这些集合被浓缩成三个独特且具有代表性的档案，每个档案分配语义属性，例如姓名和年龄。最终档案：(a) 北伟伟，(b) 亚当·特纳，(c) 维多利亚·安·克拉克，各自突出个人计算的不同侧面：档案 A 代表学生和内容创作者的环境，档案 B 代表法律执行环境，档案 C 代表高级金融分析师的背景（详细规格见图 3）。为保持一致性，我们筛查时间和语义冲突，仅在不可避免的情况下进行最小编辑；我们进一步去除系统生成的非用户工件，并使用一致的化名匿名化敏感标识符，从而生成一个适合的无索引“针堆”文件系统。

# 注释

如图4(a)所示，我们通过一个混合管道对问答进行标注，该管道结合了专家主导的手动创作和大型语言模型（LLM）辅助的合成。来自每种原型特征基础上贡献者群体的领域知识标注员创建显式用户驱动的问题，这些问题根植于他们自己的文件和日常工作中，并反映真实的个人计算需求。与此同时，专有的LLM（Comanici等，2025；PBC，2025；OpenAI，2025a）根据上下文元数据（文件路径、时间戳和目录层级）生成合成候选问题，以提高覆盖范围并平衡模式和证据分布。所有候选问题随后由人类标注员整合，他们对其进行严格审核、编辑和筛选，形成一个最终的精心策划的问题集，确保这些问题有意义、事实正确，并忠实于个人背景。我们通过意图级去重和模式级去重进一步强化多样性，同时在整个集合中平衡模式组合和证据集大小。对于每个筛选出的问题，标注员通过将其构建为一个完整的轨迹，包括逐步推理痕迹、显式文件基础证据和智能体能力标签，来构造中间监督。

# 3.2 任务

每个问答对被注释为结构化轨迹，存储为 JSON 记录，包括问题和真实标注答案、逐步推理过程以及针对内容的细粒度本地化证据和原子指针（例如，页面索引、表格单元格或文本跨度），同时包括文件级的基础元数据。每个轨迹进一步标记为智能体能力标签（图 4(c)），将所需行为分解为三个阶段：搜索（系统级导航、语义检索）、感知（文件系统理解、特定模态的理解和文本、文档、图像、视频和音频的基础），以及推理（基本推断、计算、摘要和验证）。所有轨迹均由人类注释员撰写，遵循最简约的注释原则，明确链接这些阶段，从而实现多模态智能体行为的可解释诊断。该框架支持细粒度能力分析、证据级基准测试以及在现实个性化文件生态系统中对长时间跨度、多步骤推理的评估。

分析任务。分析任务评估智能体是否能够从设备存储的文件中，通过汇聚跨时间的真实个人事实，构建一个连贯的用户级模型。每个查询针对高层次的个性特征，包括偏好和日常习惯、时间安排约束、回顾性陈述和工作流程等。与单一查询不同，分析任务需要综合来自多模态内容的异构线索、文件系统组织和时间规律，并生成全球一致的、可操作的响应。例如，“在2025年10月27日下午，为我安排一个好的计划。”测试智能体能否整合相关文件（例如日历和通信）中的证据，以及历史习惯和已表明的偏好，从而生成一个符合现有承诺的可行的个性化计划。事实保留任务。事实保留任务主要评估智能体检索、理解和推理分布在多模态文件中的事实信息的能力，这些文件存储在用户的设备上。与传统的开放领域问答不同，这些任务基于用户特定的、设备规模的上下文，要求智能体准确定位相关文件，解释异构内容类型，并整合信息以产生精确、上下文相关的答案。例如，查询“我有关于最大流问题的笔记。这些笔记记录来自哪个课程，以及课程持续时间是多少？”测试智能体能否识别、理解并推理存储在不同格式中的内容。解决此类任务不仅需要细粒度的检索和多模态理解，还要求在时间上和语义上相关文件中进行长期的上下文推理。

# 4.2 基线方法

RAG 方法。标准 RAG（Lewis et al., 2020）和 Self-RAG（Asai et al., 2024）代表经典的检索与生成管道。它们依赖于对多模态嵌入的浅层语义检索，并假设相关证据会在候选集内被呈现。这些方法缺乏多步检查、适应性搜索、跨领域合成或持久记忆能力，这些都是 HippoCamp 所必需的。 搜索智能体方法。搜索智能体基线将检索嵌入到一个明确的多轮搜索推理循环中。遵循 ReAct 框架（Yao et al., 2023），我们使用 Gemini 2.5 Flash（Comanici et al., 2025）和 Qwen3-30B-A3B（Bai et al., 2025）模型来实例化 ReAct，其中系统在推理步骤和明确搜索操作之间交替进行，后续生成依赖于已检索的观察结果。Search-R1（Jin et al., 2025）同样通过结构化标签交错推理与搜索，实现跨多个步骤的动态证据获取。 自主智能体方法。我们评估基于 Docker 的自主智能体，这些智能体实例化了多个模型后端，包括 Gemini 2.5 Flash（Comanici et al., 2025）、ChatGPT 5.2（OpenAI, 2025b）、Claude Sonnet 4.5（PBC, 2025）和 Qwen3-VL-8B-Instruct（Bai et al., 2025），以及专有的产品级智能体模式。在主要表格中，我们报告了 ChatGPT 智能体模式（OpenAI, 2025a；OpenAI et al., 2024）；由于严重的文件系统和长文档处理失败，Claude 智能体（PBC, 2025）设置单独讨论。这些系统通过递归工具使用进行操作：发出查询、查看文件预览、解释中间结果并更新假设。它们代表了公开可访问的最先进的扎根推理范式，尽管它们的工具策略和架构假设有显著差异。

# 4.3 评估协议

为了严格评估智能体在个性化多模态信息管理中的性能，HippoCamp 采用了一种评估工具，覆盖两个核心维度：问答质量和证据检索准确性。每个维度通过自动指标和基于大型语言模型（LLM）的判别任务进行评估，所有方法都通过一个公共评估接口在特定的个人配置访问约束下进行支持，支持三种执行模式：本地检索、虚拟 Docker 终端智能体和托管商业智能体模式。执行模式、预算和稳健性检查的具体细节在 D 节提供。问答评估方面，对于所有任务，我们采用 LLM 作为评判者（Li et al., 2024a）的方法。一个强大的参考 LLM 被提示输入问题、真实答案和模型生成的响应，并被指示生成一个二元正确性判断（是/否），以及在标准化的 0-5 评分标准上给出质量评分。提示明确要求评判者考虑事实一致性、推理合理性和上下文个性化。我们报告整体准确性，以响应中正确答案的比例来衡量，使用召回率和基于真实证据集的 F1 分数。F1 分数捕捉相关性与非相关性的平衡，同时真实评估智能体是否获取了支持正确推理的所有必要证据。

# 4.4 实验结果

表2报告了所有配置和两种任务类型的结果。总体来看，目前的方法与人类专家之间仍存在较大差距，暴露出RAG管道和智能系统在长期个性化文件推理方面的基本弱点。这些结果促使HippoCamp作为一个诊断基准，揭示多模态检索、分层文件理解和基于证据的多步骤推理中的失败，并表明需要更强的索引、持久内存和以验证为中心的智能体架构。接下来，我们按方法类别分析性能。RAG方法。整体来看，RAG管道的表现较差，尤其是在配置方面，性能依然低下（例如，标准RAG：18.4 F1 / 26.7 准确率；自我RAG：15.2 F1 / 10.0 准确率）。检索较为脆弱，常常返回浅显或不相关的文件，而生成缺乏稳健的跨文件聚合。这在配置(c)维多利亚中尤为明显，其中自我RAG未能产生任何正确的配置答案（准确率为0.0）。

# 5.2 系统性故障分析

实体错误归属。即使证据真实且相关，将其绑定到正确的指称对象依然是一个独立的失败阶段。最强大的系统可以检索到真实的健康记录，却将它们分配给错误的实体。ChatGPT代理模式能够定位真实的健康相关记录，但将例行事务归因于用户的宠物而非用户，这暴露了在多个实体共享同一文件系统时进行第一人称约束解析的难度。其检索准确性始终优于理解能力（表3中搜索准确率为$56.5\%$，感知准确率为$28.5\%$）。这种影响依赖于个人档案：在亚当的文档中心法律档案中，其事实准确率达到$90.3\%$，而在贝的档案中仅为$31.2\%$，较为丰富的人际上下文增加了实体歧义（见表2）。评估方法未包括明确的最终阶段检查，这些检查确保生成的答案依赖于一个最小的、连贯的证据集。结果在表3中F1准确性差距中显而易见：检索和推理的F1始终超过答案准确性，这表明在早期阶段引入的错误——错误绑定、幻觉或不足的辨别力——未能得到控制，传播到最终输出。成功模式：迭代发现。ChatGPT代理模式展示了一种质的不同策略，通过迭代的系统探索。通过反复查阅目录和阅读候选文件，它逐步完善假设，并可以从早期失误中恢复，从而产生最佳的整体表现（在表2中档案准确率为48.3%，事实准确率为$62.8\%$）和最平衡的能力档案（见表3）。在结构化领域的提升最为显著，亚当的档案准确率达到$55.0\%$，事实准确率为$90.3\%$（见表2），但在跨模态档案中，由于信号较弱且异质性更强，仅达到$35.0\%$的档案准确率。它的优势不在于单次检索的强度，而在于能够在多个阶段重访和纠正错误——从偏离目标的查询中恢复、完善证据假设，并逐步缩小支持集的范围。

# 5.3 展望：设计下一代文件系统智能体

评估超越静态角色的真实个性化。与依赖显式静态角色的先前基准不同，HippoCamp 证明了在本地数字环境中，真实个性化本质上是一个多模态跨文件推理挑战。为了真实地建模个体用户的数字生活，智能体必须综合散布在异构文件类型中的隐含行为信号，而不仅仅是检索显式陈述。此外，它们还必须成功地将账户持有人与周围其他实体区分开，以避免关键的错误归因，同时还需对用户的数字足迹进行长期的时间连续性推理。HippoCamp 要求对这些复杂、不断演变的约束进行纵向综合，从而超越通用检索，提供对智能体在个性化计算生态系统中真实适应能力的严格评估。如何设计好的文件系统智能体？人类能力与智能体性能之间的差距（表3）表明，强大的文件系统智能体需要更加紧密地结合结构交互与验证。具体地，智能体应将文件系统层次和跨文件关系视为搜索的归纳偏置，用迭代的信息觅取取代一次性检索，通过元数据和轻量级检查进行引导，并集成证据中心验证循环，将中间推理与具体文件路径和局部证据绑定，以减轻幻觉产生（图6）。结构感知搜索。文件层次、时间规律、附件关系和版本历史编码了纯嵌入相似性所丢弃的组织意图。我们的结果显示，仅依赖语义检索的方法常常混淆主题相似但语境相关的文件（第5节）。将文件系统结构视为搜索的归纳偏置——而不是将文件膨胀为单一向量索引——可以在管道的早期阶段减少目标外检索。答案生成前的证据缩小。表中检索 F1 和答案准确性之间的一致差距表明，在检索相关文件时很难确保答案的准确性。虽然检索模型在文件检索上取得了最高的 F1，答案准确性却是最低的（表3），这表明没有后续证据选择的广泛候选池引入了噪声，而下游推理难以过滤。智能体应在生成答案之前形成一个最小、充分的支持集，而不是对所有检索出的候选项进行总结。个人本地实体建模。个人文件系统包含多个共现的实体——账户持有者、家庭成员、宠物、同事——它们共享文件和上下文。图6 中的实体错误归属失败表明，智能体必须显式维护和更新引用模式，而不是假设所有第一人称引用都映射到单个用户。没有这样的建模，正确的检索和感知仍然会产生错误的答案。显式阶段的验证。评估方法包括验证阶段，将答案绑定到局部、文件级的证据，并检查内部矛盾。增加这样的阶段将直接解决第节中识别的验证不足，并有助于缩小检索质量与答案质量之间的差距。一个强大的文件系统智能体不仅必须广泛检索和跨文件综合，而且还必须确认其响应可追溯到最小的、有条理的支持集。

# 6 结论

本研究介绍了HippoCamp，一个评估智能体在搜索、感知和推理现实多模态个人文件系统能力的基准。我们的分析揭示了主要瓶颈不在于证据检索，而在于检索后的流程——证据区分、多模态关联、实体绑定和最终验证——并且，分析需求需要与事实保持不同的能力组合。个性化多模态记忆仍然是一个未解决的挑战。HippoCamp为诊断这些不足提供了严格的基础，并指导下一代个人文件系统智能体的发展。

# A.1 参与者池和来源选择

HippoCamp 是通过对 100 余名个人设备用户的访谈构建而成。本节具体说明了在进行特征聚合之前形成候选池所采用的参与者筛选和来源选择协议。招募涵盖不同年龄、生活状况和技术背景的用户，并在人口统计学、社会经济、职业和生活方式等维度上进行分层筛选，以保持多样性，同时保持设备使用强度的可比性。我们仅保留满足以下可复现标准的参与者： (1) 文件系统丰富性和模态覆盖。设备必须包含一个稠密的、异质的语料库，至少有 500 个用户文件，涵盖至少五种模态中的四种（文本、文档、图像、视频、音频），并且至少有 10 种不同的文件扩展名。 (2) 纵向深度。语料库必须涵盖至少三个月的创建或修改活动，参与者必须报告以个人工作站或笔记本电脑作为主要设备的持续全职学习或专业实践，以产生稳定的例行程序和重复的工作流程。 (3) 证据的完整性和可审计性。候选来源必须支持可审计的用户级推断，具备最低限度的证据清单：(i) 针对关键个人事实的跨文件证实（至少两个独立支持的证据），(ii) 长期行为的一致时间锚（时间戳或日期记录），以及 (iii) 可解释的组织痕迹（目录结构、命名约定或重复记录），便于重构日程、例行和工作流程，而无需依赖不可验证的叙述。此标准排除在构建分析时无法凭证据支持的“剖面漏洞”。我们还排除了被僵化的企业 IT 模板或集中管理的目录方案主导的来源，这些方案会模糊用户驱动的组织。未满足任何标准的候选人会在聚合之前被移除，剩余的贡献者形成用于附录 A.3 的筛选候选池。最后，在进行任何面向发布的处理之前，我们获得明确的同意并采取隐私保护措施：参与者在受控处理下提供匿名的目录树和代表性文件，敏感标识符被删除或进行pseudo化，同时保留证据线索（见 A.4 节）。

# A.2 深度访谈协议

# A.3 原型聚合

我们通过聚合来自多个贡献者的文件构建三个原型档案。在本附录中，连贯性意味着每个生成的档案形成一个逻辑封闭且行为上合理的环境，其中时间戳、实体和跨文件引用相互一致，因此，剖析和事实保留查询都能够提供可验证的文件基础证据。为满足该要求，聚合过程分为四个阶段：保持分布的分区、一致性检查、最小编辑修复和最终人工验证。贡献者的分区根据模态组成、文件类型频率和高层次的组织模式进行选择。分区程序旨在保持长尾格式覆盖、目录结构多样性，以及个人、学术和专业材料之间的相对平衡，而不是将特定格式或工作流程集中在单一档案中。当多个分配满足这些约束时，我们优先选择在时间范围、项目结构和重复使用模式上具有更强内部兼容性的分配。一致性检查。在贡献者分区后和进行任何最小修复之前，我们应用自动化和人工验证过程进行一致性检查：验证跨时间戳的文档（例如日历、电子邮件、日志和媒体元数据），以避免不兼容的时间线；(i) 实体和身份的一致性，以防止重复名称、标识符和持久实体之间的冲突，这样会造成矛盾的叙事；(ii) 项目和工作流程的一致性，以确保多文件线程（例如课程材料、案件文件和分析）内部保持一致，具有有效的跨文件引用和依赖关系。检测到的违规行为通过源重分配解决，或者在必要时，删除最少数量的冲突条目。最小编辑政策。编辑仅在需要修复矛盾、否则将破坏档案级一致性时引入。典型的干预措施包括化名对齐、移除重复或冲突的标识符，以及在恢复一致性时对元数据或文件名进行有限规范化。当无法在本地修复冲突时，我们将删除最少数量的违规项。人工验证。最后，聚合后的档案作为最终候选环境进行审核，以验证跨文件一致性、代表性结构以及不存在未解决的隐私或一致性违规问题，然后发布基准。贡献的注释员随后提供一揽子签字，以确认组装的环境保留了原生的组织模式和习惯使用信号。这些最终检查生成三个连贯的原型档案，适合后续的质控构建和基于证据的评估。

# A.4 隐私、过滤和匿名化

我们采用隐私优先的数据治理协议，旨在消除个人身份信息（PII）的泄露，同时保留进行基准评估所需的证据线索。参与是严格的自愿选择，需要明确同意，参与者保留在任何时间撤回其数据的权利；一旦撤回，相应的文件将从候选池以及从任何基于这些文件的汇总档案中移除。所有操作均在受控访问下进行，未经隐私处理和参与者验证，未使用原始数据进行注释或分析。我们过滤系统生成的、非用户创建的艺术品，这些艺术品不反映用户意图（例如，操作系统缓存、应用程序临时文件、索引数据库、回收站残留物以及其他后台瞬态文件）。通过一套可复现的规则集实现过滤，结合基于路径的模式（例如，操作系统缓存目录）、文件类型和文件名启发式（例如，已知的缓存扩展名）以及重复/近重复检测。这一步骤在处理用户生成内容时比较保守：即使是小型或稀疏的普通文件我们也会保留，仅排除那些显然是操作系统或应用程序生成且在语义上不承载内容的工件。

在保留证据的前提下进行匿名化处理，基准数据使用专有图像生成模型（Nano Banana (Comanici et al., 2025)）从虚构身份配置文件生成或复制，以保持任务结构，同时删除标识符。任何外部来源的非敏感资产仅在允许重新分发和商业使用的许可下包含；不保留任何包含现实世界标识符的原始个人媒体。我们通过删减和一致的化名处理来匿名化敏感标识符，同时保留进行证据验证所需的最小线索。具体而言，我们 (i) 用在配置文件内一致的稳定化名替换个人识别信息，包括姓名、电子邮件、电话号码、地址、身份证件、账户名和组织名称，(ii) 删除或清理可能揭示身份的嵌入元数据字段（例如，作者字段、设备识别符、GPS 坐标等），以及 (iii) 保留评估所需的结构性和时间性信号，例如目录层次、相对时间戳和排序，以及交叉引用。当进行删减时，我们保持格式一致性（例如保持日期和数字模式），以避免影响后续验证任务，并确保匿名化不会产生虚假的证据。参与者的活动和最终批准遵循参与者的要求。贡献者检查处理后的输出（目录树和删减文件），并批准未保留敏感信息且剩余内容对其设备上下文的可接受表示。只有通过此参与者检查的数据才能进入后续聚合和注释。通过过滤、匿名化和参与者验证，确保 HippoCamp 不包含任何可识别的个人信息，同时保留进行有据评估所需的证据线索。

# A.5 用于数据增强的外部基准

在配置文件聚合（第 A.3 节）后，我们观察到一种生态不对称性：由于保密限制（例如，监管披露、客户合同和财务报表），某些在专业工作流程中频繁出现的特定文档形式在个人设备上自然处于代表性不足的状态。这种稀疏性在以文档为中心的配置文件（b）亚当（法律）和（c）维多利亚（金融）中最为明显，它可能减少依赖于这类材料的现实任务模式的覆盖率。为了在不妥协隐私或连贯性的情况下改善覆盖率，我们为这两个配置文件引入有限量的经过策划的公共领域源材料。具体而言，我们使用基于 FinanceBench 的文档作为金融资料，以及基于 LegalBench-RAG 的文档作为法律资料，以实现文档形式的丰富，而不是直接合并基准。任何导入的内容都会被重写/消毒，以符合 HippoCamp 的虚构身份系统（第 A.4 节），所有生成的问答和轨迹都会在我们的框架下重新标注，附上完整的上下文元数据（例如，文件路径、时间戳和目录层次信号）。这些材料仅用于补充在文档形式上代表性不足的内容；基准语义、任务构建和最终注释仍由 HippoCamp 流程管理。 FinanceBench 集成（Islam 等，2023）。FinanceBench 提供分析师相关的文档类型。我们将其文档纳入金融配置文件，作为额外的证据性文物。当使用 FinanceBench 的问答项目时，严格视为筛选后的候选材料，并重写或调整为与虚构角色和 HippoCamp 任务分类法一致的以用户需求为驱动的问题，而不是逐字重复。所有保留的问题都会配有新构建的逐步推理、局部证据和文件基础，元数据规范化为我们的轨迹格式。 LegalBench-RAG 集成（Pipitone 和 Alami，2024）。由于其原始实体和问答与真实司法管辖区和案件标识符关联，LegalBench-RAG 需要更重的适配。因此，我们仅在筛选后将其问答用作灵感或种子候选，并系统地重写问题和消毒文档内容，以去除现实世界实体，同时保留与任务相关的法律结构。目标是保留现实的法律文档结构，而不是保留原始基准的实例。与重写任务无关的文档会被移除，剩余的语料库会被整理为完整的时间一致性，以符合虚构的法律角色（第 A.4 节）。所有保留的问题都将在 HippoCamp 的轨迹框架下从头开始重新标注，包括完整的元数据、证据定位和逐步推理痕迹。 分布和一致性保护。为了避免基准因增强来源而产生偏差，我们限制外部来源文档的贡献，使得配置文件级别的模式和文件类型统计保持在聚合过程中的分布约束内（第 A.3 节）。所有增强项在纳入之前都会经过与本地源材料相同的一致性检查和人工验证。这些外部来源的文档仅构成整体语料库的少数补充，目的仅在于改善覆盖率和在代表性不足的文档领域中的结构完整性。因此，任务语义、注释协议和评估目标仍由 HippoCamp 定义，而不是继承自源基准或其原始标签空间。

# B 注释方案与质量控制

HippoCamp 的注释并不是孤立的问答对，而是作为有依据的轨迹，将问题和答案与文件系统的基础、局部证据以及逐步推理轨迹结合在一起；在本节中，我们详细介绍轨迹模式、证据定位与原子单元、参与者在环的问答构建流程，以及带有质量控制的注释者之间的一致性。

# B.3 领域语义与解释规则

我们阐明了几个在正确解读发布的模式和重现下游分析中重要的领域级约定。字段 fle_path 记录了回答查询所需的真实标注数据的最小文件集，而不是在数据构建过程中标注者遍历的完整文件集。理论上，它代表了推导黄金答案所需的最小标注支持集，而 flenumber 则作为其基数指示符。字段 file_modality 和 evidence-level modality_type 在不同的粒度层面上定义。file_modality 是对 fle_path 中列出的支持文件的集合值摘要，而 modality_type 在归一化后分配给每个本地化证据项，表征证据单元本身。因此，前者应被视为文件级汇总，而后者提供项级的模态信息，可能捕捉到比支持文件整体所表达的更细粒度的结构。答案被定义为针对目标的自然语言响应，而不是推理过程。它以直接可判断的自然语言书写，根据需要可包含一个或多个句子以确保完整性，并可以选择性地在方括号中包含文件引用（例如，[meeting_notes.pdf]），当这样的引用有助于消歧义时。然而，答案设计上不应包含思考链、逐步推导或不必要的解释性叙述。最后，agent_cap 编码了解决实例所需的能力，而不是模型预测或观察到的系统行为。在该字段中，agent_cap.reasoning 遵循一个约束标签系统：basic 表示最小的推理状态，并与所有更强的推理类别互斥，而 computation、verification 和 summarization 可以单独或共同赋值。因此，basic 的存在排除了任何更高阶推理标签的共存，而任何需要一个或多个更高阶推理操作的实例必须排除 basic。

# B.4 证据对象规范

每个证据条目都是一个本地化的支持对象，旨在直接与源文档进行核对。至少，一个证据对象包含 evidence_id、file_path、标准化的项目级 modality_type、evidence_text 字段，以及一个或多个在 evidence_locator 下的定位条目。字段 evidence_id 在单个轨迹记录中是唯一的，并作为推理步骤引用支持证据时使用的关键。它不需要在整个基准发布中是全局唯一的。字段 evidence_text 的目的是以适合验证的形式保留支持答案的内容，而不是总结整体。对于文本或文档证据，它通常是直接摘录的片段。对于视频，内容通常是精确的转录，以实现在本地化段落中的表现。对于图像类证据，它是锚定于本地化区域的简洁内容描述，而不是自由形式的解释。本地化存储在 evidence_locator 中。从概念上讲，每个定位器表示为 {unit, position} 对，其中 unit 定义了测量基础，position 指定了实际范围或索引。典型的文档证据使用基于页码的单位，而时间媒体使用基于时间戳的范围。对于基于页码的文档，位置可能包括系统页和打印页；对于音频或视频，位置通常是起始和结束的时间戳范围。

# B.5 理由追踪规范

每个条目在 rationale 中表示一个注释的推理步骤。一个合理的步骤至少包含一个 step_id、一个自然语言的 rationale_text 和一个指向支撑该步骤的证据项的 evidence_id 列表。step_id 在包围的合理轨迹中是唯一的；rationale_text 描述该步骤中的中间操作、观察或结论；而 evidence_id 提供了明确的支撑链接回到局部支持。发布的合理轨迹遵循一个规范化的三阶段结构。第一阶段是规划，它将任务分解为子目标或搜索意图。第二阶段是导航和阅读，它涵盖了文件发现、区域定位和证据获取。第三阶段是整合与验证，它在不同来源之间综合证据，执行任何必要的检查或计算，并组装最终的有根据的答案。这些阶段是概念性规范化类别，而不是严格的用户可见标签，但它们为推理分析提供了一致的过程模板。在步骤级别上强制有根据性。我们仅允许在定位证据之前为抽象规划步骤提供 evidence_id = [ ]。任何报告文件内容、提取事实支持、执行跨来源整合、验证主张、基于源材料执行计算或直接贡献最终答案的步骤都必须引用一个或多个有效的证据 ID。换句话说，空的证据引用仅允许用于高层规划，而不适用于阅读、整合、验证或承载答案的步骤。

# B.6 架构验证规则

为了确保可重复性，我们对发布的记录应用了一种轻量级的模式验证协议。该协议的目的是不强制实施过于严格的归一化层，而是保证发布的 JSON 对象始终可以解析，并且下游评估所依赖的记账在不同实现中稳定且可重复。我们的验证程序检查以下条件： - 证据-引用一致性。每个由理由步骤引用的非空 evidence_id 都必须解析为同一轨迹记录中相应 evidencel] 数组中定义的有效 evidence_id。 - 文件系统可解析的支持路径。每个顶层 fle_path 条目以及每个证据级别的 file_path 都必须能够解析为发布的文件系统中现有文件。 - 定位器范围有效性。每个证据定位器必须符合其声明的单位。对于基于页面的证据，页面索引必须位于相应文档的范围内；对于基于时间戳的证据，开始和结束时间必须定义一个在相关媒体文件时长内的有效区间。 - 答案-证据支持一致性。注释的答案必须可以由链接的证据集支持。我们通过手动审计和脚本级别的合理性检查对证据引用及其局部内容进行评估。在这些约束下，解析器可以通过读取顶层任务和支持字段，解决 evidence] 为以 evidence_id 为索引的证据映射，依据引用的证据为每个理由步骤提供依据，并验证结果结构与上述验证规则的一致性。该程序足以重建能力分析、难度计算和证据级匹配指标所需的记账。

# B.7.1 动机

个人文件问答中的一个核心挑战是证据在不同模态下具有自然的局部化特性，且粒度各异。例如，文档通常在页面级别上进行引用，而音频和视频则在时间戳级别上进行引用，图像和嵌入的视觉区域可能需要空间定位。如果没有一个模态对齐的证据粒度概念，评估将在模态之间难以比较，并且可能会高估检索质量，尤其是当一种方法识别了正确的文件但未能定位到确凿证据时。为了解决这个问题，我们引入了原子单元（AUs），这是一种模态规范化的抽象，表示用于细粒度定位分析和错误诊断的最小证据承载区域。从直观上看，AUs 提供了一个诊断接口，不仅用于询问智能体是否找到正确的文件，还用于查询其答案是否基于该文件的正确部分。

# B.8 AU生成程序

AUs是通过特定模态的程序生成的，这些程序在精确度和可重复性之间进行权衡。文本。对于自由格式文本，我们将连续的词元或句子跨度视为AUs。在发布的数据中，证据以evidencetext的形式存储；AU映射是通过将注释的跨度对齐回源文件中的相应文本区域来获得的。文档。对于分页文档，默认的AU是页面等效单元，因为页面索引是记录在evidence_locator中的本地定位器。需要更细结构时（例如，表格行、图形标题或嵌入的视觉区域），可以使用页面坐标和可选的表格单元或图形区域细分将页面进一步细化为文档区域。这种页面优先的设计与当前发布匹配，同时保持对更精细文档基础的可扩展性。图像。对于图像，AUs是在空间坐标系上定义的。一个实用的实现是一个规则的补丁网格（例如，在图像平面上的$1 6 \times 1 6$补丁），在有具有意义的人类区域时，可选聚合成边界框。在当前发布中，图像证据通常以文本形式表示，通过evidence_text；因此AU分配主要作为分析时的抽象，除非额外注释了显式空间区域。视频。对于视频，AUs是依据固定采样规则得出的时间块或帧。一个可重复的默认值是以预定义帧速率均匀采样（例如，分析时1fps）或具有固定步幅的时间戳窗口分割。当前发布使用基于时间戳的定位器，因此AU映射直接由引用的时间间隔诱导。音频。对于音频，AUs是由时间戳窗口诱导的时间段。标准实现使用固定长度的帧或短段（例如，具有恒定跳幅的分析窗口），但发布的基准在时间戳级别记录证据；因此，相应的AU是由注释时间间隔跨越的段。

# B.9 AU-证据映射

HippoCamp中的每一条证据项目都包含一个人类可读的证据定位符，而AU映射则确定性地将该定位符转换为一个或多个与模态对齐的单位：文档页面定位符对应页面等效的AU；文档区域/表格/图形定位符对应页面内部的区域级AU；图像区域空间补丁集或边界框AU；时间戳或时间戳范围的音频/视频时间AU跨度。为了支持可靠的评估，AU匹配在适当情况下定义了容忍度。对于时间媒体，匹配可以允许标注区间周围有小的时间戳松弛；对于空间区域，可以通过标准区域标准（例如，当边界框可用时的IoU式重叠）来测量重叠。这些容忍度确保AU级别的基础保持可重复性，而不会因微小的对齐差异而变得脆弱。

# B.10 评估、诊断和训练中的使用

AUs 在 HippoCamp 中有三个目的。评估。在评估时，AU 标准化的基础提供了一种模态一致的方法来评估一种方法是否定位了决定性证据，而不仅仅是检索了正确的文件。对于多模态文件，这一点尤为重要，因为基于文件的命中可能会产生误导。因此，AU 感知比较作为文件级检索指标的诊断补充，提供了更细粒度的证据重叠和基础分析，在定位质量重要的情况下尤为有效。诊断。在诊断时，AUs 有助于区分不同的失败模式。例如，一个模型可能识别出正确的文件，但在错误的页面、错误的时间戳或错误的视觉区域进行定位。这类错误在仅依赖文件级评分时是不可见的，但通过 AU 级分析可以暴露。这种意义上，AUs 的引入主要是为了支持细粒度的感知和基础错误分析，而不是定义独立的标题分数。训练与发布考虑。在当前基准发布中，AUs 主要支持评估和诊断，而不是作为独立的训练监督目标。因此，AU 级信号主要用于细粒度分析和诊断细分，而基准的主要报告检索指标仍基于文件级。这一设计受到隐私、版权和注释成本考虑的驱动，尤其是针对空间上本地化的个人媒体。同时，AU 公式仍与未来的隐私保护派生监督机制兼容，例如掩蔽裁剪、区域描述符或仅基于 AU 的表示方式，这些方法在不暴露原始个人内容的情况下保留了基础结构。

# B.12 双源问题提案

手动提案。HippoCamp 问题的主要来源是参与者和熟悉特定档案系统的专家注释员的手动创作。这些问题显然是由用户驱动的：它们源于参与者在日常个人计算中可能遇到的具体信息需求，例如回忆过去的事实、重建工作流程、总结先前的事件或在当前约束下进行规划。由于贡献者理解自己的习惯、组织策略和重复任务，因此手动提案能够捕捉到独特的真实意图，这些意图仅靠文件本身难以恢复。合成提案。为补充手动创作，我们使用大型语言模型生成候选问题，这些问题受到限制的上下文元数据的条件影响，例如文件路径、时间戳、目录结构和种子示例。合成提案的角色不是定义基准语义，而是提高在表现不足的维度上的覆盖率，包括模态组合、证据集规模和长尾任务模式。因此，这些提案严格视为候选项，并在整合阶段接受审查、修订、合并或拒绝。

# B.13 候选人整合与筛选

所有手动创建的候选项和大语言模型建议的候选项经过人工筛选阶段进行整合。在此阶段，标注者保留符合我们的构建时间要求的候选项： (i) 真实性，意味着一个合理的答案可以由档案中的实际文件支持；(ii) 非平凡性，意味着查询需要有意义的检索、感知或推理，而不是直接查找文件名或元数据；(iii) 低冗余性，意味着候选项在意图或推理模式层面上不重复现有项目；以及 (iv) 隐私安全，意味着措辞和预期证据要求不会重新引入超出匿名档案的敏感信息。保留的集合形成了传递给去重、平衡和轨迹结构化的候选池；最终发布时的接受标准在附录B中单独描述。

# B.14 去重与覆盖约束

为了避免在少数查询形式中出现过度集中，我们在两个层面上进行了去重和平衡。意图级去重。表达相同潜在信息需求的问句，仅通过表面上的措辞改变需要合并，并保留一个代表性的表达形式。模式级去重。我们还抑制依赖于基本相同解决结构或证据配置的重复问题，即使这些问题的措辞不同。例如，两个问题要求相同的解决方案和相同的推理模式被视为相同项。覆盖平衡。在去重后，我们重新平衡保留的候选集，以改善不同模态组合、证据集大小和任务类别的覆盖范围。特别地，我们监控相对相关性，以确保最终基准既包含常见的个人计算问题，也涵盖更具挑战性的长尾案例。

# B.15 轨迹结构化

对于每个保留的问答候选项，人类标注者构建用于基准发布的真实轨迹记录。该记录包括 (i) 最小支持集和文件级元数据，(ii) 带有明确定位器的证据对象，(iii) 分步推理轨迹，以及 (iv) 覆盖搜索、证据感知和推理的能力标签。该阶段的目标是将一个接受的问题转换为一个紧凑的、符合模式的支持结构以供评估和后续诊断。一个关键设计原则是极简主义的黄金轨迹。目标不是列举每条合理的解决路径，而是记录足够证明答案的最小证据节点和推理过渡。因此，生成的轨迹定义了一个紧凑的评估支持结构，同时为智能体在推理过程中发现更长或替代的有效路径留出了空间。

# B.16 模型辅助协议

为了保持大语言模型（LLM）助手的可重复性和界限性，我们限制模型可见的信息。LLM 只能观察有限的上下文元数据和选定的示例，包括文件路径、时间戳和目录层次提示；它们不能无限制访问完整的文件系统、原始个人标识符或任何外部知识来源。模型被明确指示不得虚构个人信息，不得引入不支持的事实，也不得提出依赖于提供的语料库外部信息的问题。典型的输出是候选问题或轻量级 JSON 骨架，随后由人类标注者进行编辑和完善。在该协议下，模型仅用于边界提案或轻量级结构支持；所有最终的基准内容仍需经过人工修订和批准。

# B.17 LLM辅助提案的提示家族

# B.18 标 annotator 一致性与质量控制

本节描述了在候选构建和轨迹创作之后所应用的协议、裁决和发布时质量控制程序。这里的重点不是基准条目的提议或结构，而是已接受记录在发布前如何审查一致性、隐私安全、真实度和模式合规性。

# B.18.1 注释者设置与采样协议

HippoCamp由了解领域的人工标注者进行标注，这些标注者来自于支撑原型数据集的贡献者群体。考虑到组织逻辑、持续工作流程以及他们自身数字环境中的实际规范需求，他们能够制定和验证基于真实场景的任务，而这些任务仅通过外部标注很难恢复。然而，为了减轻特定档案的偏见，所有记录在后续都经过交叉检查、二次审核和裁决。为了进行一致性和验证分析，我们采用了覆盖所有三种档案的分层抽样协议，同时涵盖两个任务类别（事实保留和档案记录）、不同的模态配置和多种难度层次。这确保了可靠性评估不仅在简单的文本主导案例中进行，也在多模态和长时程实例中进行，这些实例对于智能体评估具有最强的诊断能力。

# B.18.2 统一评审协议

所有标注者在二次审核和发布准备过程中遵循统一的审查协议。对于每个被接受的记录，审核者验证在发布的匿名环境下问题的可解释性，以及相关的标签、任务标签和理由字段是否保持内部一致。审核者并不寻求枚举所有可能的有效推理路径；相反，他们验证发布的记录是否连贯、最小充分且符合模式要求。对于每个接受的问题，标注者识别出仅使用严格必要的文件和证据范围来证明黄金答案的最短合理解决路径。最终的轨迹记录了领域专家采取的最直接、可信的人类推理路线，仅包括关键文件、相关页面或时间戳等基本要素，以及最少的支持证据。这种最简化的设计在评估过程中保持了轨迹的多样性：智能体可以遵循更丰富或更长的有效路径，但任何成功的解决方案必须至少恢复记录在黄金轨迹中的基本证据节点。因此，发布的轨迹应被解读为一个紧凑的支持结构，而不是详尽的执行轨迹。

# B.18.3 裁决协议

注释分歧通过裁定而非自动多数投票来解决。当两个注释出现冲突时，项目会转交给额外的审阅者，后者会重新检查源文件、证据链接以及任务规范。如果分歧反映了底层内容的模糊性，注释者会通过集中审查来解决，并在必要时修订证据范围或问题措辞，以恢复可解释性和基础性。因此，最终发布的记录是经过裁定的金标准注释，而非原始投票汇总。裁定在特征任务中尤为重要，因为分歧更常来自于纵向解释而非简单提取，相应的注释即使在逻辑上 plausible，若遗漏了限制性条款、数值约束或领域特定例外，也可能仍然不可接受。

# B.18.4 质量控制与自动化合理性检查

我们采用多阶段发布时质量控制流程，结合了基于规则的检查、人工审查、个性一致性验证以及数据集层面的平衡。接受标准。以下接受标准适用于考虑最终发布的记录，而不是附录 B.13 中描述的早期候选筛选阶段。每个候选实例必须满足五个标准：（i）基础性，意味着答案完全由提供的文件支持，而不依赖于不可验证的外部知识；（ii）清晰性，意味着这个问题对于仅能访问已发布文件的第三方是可解释的；（iii）任务现实性，意味着它反映了相应个体的合理信息需求；（iv）非平凡性，意味着它不能通过表面元数据查找或简单复制来解决；（v）隐私安全性，意味着没有任何文本组件重新引入超出已批准匿名档案的标识符。自动验证。除了人工审查，我们会对所有记录进行自动化的合理性检查。这些检查验证存在性和路径有效性、定位器有效性（例如页面范围和时间戳边界）、证据引用的闭合性、通过相似性阈值检测近重复问题的能力，以及基于规则扫描名称、电子邮件、电话号码、身份证和其他敏感标识符的第二次隐私审核。过滤结果。基于此流程，每个候选项要么被接受，要么被修订，或被拒绝。接受的项目满足所有标准；修订的项目包含可纠正的问题，例如模糊措辞或基础性不完整；被拒绝的项目则包含需要实质性重写的结构性缺陷，包括虚构证据、隐私泄漏或过度冗余。

# B.18.5 事后审计与最终验证

在标注和裁决之后，我们进行额外的分层人工审计，涵盖简介、任务类型、模态组合和难度水平。该审计的目的是验证发布的基准在新一轮审查下仍然保持平衡和一致，并识别任何未被早期过滤的残留标注伪影。当发现失误时，相应的记录将被修订并版本化，以确保所有事后更正都是可追踪的。最后，在发布之前，所有被接受的记录必须通过架构级验证：证据范围必须正确引用，推理步骤必须符合轨迹规范，答案格式必须与预期任务匹配，任务/简介标签必须内部一致。这些程序共同确保HippoCamp的发布是可审计的、内部一致的，并且适合进行长期多模态智能体的评估。

# D 评估协议与鲁棒性

我们在一个受控的、孤立的配置下评估 HippoCamp，该协议旨在平衡公平性、可重现性和对每种方法本土交互范式的忠实度。我们不是强行将所有方法置于一个单一的人工预算下，而是在其本土实现或服务环境中，按照预先指定的、适合方法的预算评估每个系统。本文附录围绕四个组成部分组织：共享评估约束和框架设计；方法执行模式和工具；预算、重试和随机性控制；以及用于答案级和证据级评估的指标和稳健性程序。最后，我们提供了一个扩展的指标总结，补充而不是重复正文中的结果。

# D.1 共享评估约束与框架

统一评估框架。所有评估的方法均通过共享评估工具执行，该工具标准化数据集加载、方法调用、结果记录和指标计算。这种设计的目的是不消除方法的异质性，而是确保系统在相同的基准记录、配置本地信息边界和输出规范规则下进行评估。如图5所示（在正文中重现），评估器加载基准JSON记录，通过通用接口将其分发到注册的方法后端，收集答案和可选的跟踪信息，并计算标准化指标以便于聚合和分析。

# D.1.1 流水线概述

评估流程分为四个阶段。首先，评估器从 JSON 加载每个数据集记录，提取查询文本、黄金答案以及任何可选的监督信息，例如文件级证据注释或能力标签。其次，它构建一个统一的请求对象，并通过共享评估接口将其分发到选定的方法。尽管它们的底层工具界面和调度策略各不相同，但这一评估器侧的契约在检索原生系统、终端风格智能体和托管智能体模式中是通用的。第三，评估器记录方法输出，包括最终答案，以及在可用时的检索证据、搜索轨迹、工具调用和运行时统计信息，例如延迟、步骤计数和重试次数。这些输出被规范化为固定的结果架构，以确保下游评分与方法无关。最后，评估器计算启用的指标，并生成两级输出：包含所有查询级统计信息的每查询评估记录，以及包含数据集级结果的汇总文件。设计确保了所有评估方法之间的一致性，同时支持在不同指标配置下高效的大规模重新评估。

# D.1.2 个人资料隔离评估

所有评估在严格的配置文件隔离环境中进行。对于每个查询，方法仅可以访问对应HippoCamp配置文件的模拟文件系统；在推理时，其他两个配置文件中的信息不会被暴露。此隔离适用于原始文件、可访问的元数据以及任何基准提供的接口。方法只接收自然语言查询作为输入，并必须使用在单一用户设备环境中合理可用的信息来解决该查询。该设计防止了跨配置文件泄露，并确保性能反映特定配置文件的搜索、感知和推理，而非基准范围的记忆或用户间的意外转移。

# D.1.3 执行接口

评估工具支持两大类系统。第一类是检索本地方法，包括标准的RAG和自我RAG，这些方法通过检索接口在个人本地语料库上操作，并返回基于证据的答案。第二类是互动智能体方法，包括Search-R1、ReAct变体、Terminal Agent变体和ChatGPT Agent模式，这些方法可以迭代搜索、检查文件、调用工具并在生成最终答案之前细化中间假设。无论实现风格如何，所有方法都统一规范到一个共同的评估接口：主要输出是最终答案，并且在可用的情况下，我们还记录检索到的文件或证据列表、中间跟踪或工具调用，以及运行时统计信息，如延迟、搜索迭代次数和重试次数。这种规范化使得在共享基准和个人本地访问限制下，可以对具有明显不同架构的系统进行任务级比较，同时保留它们的本地交互模式。

# D.1.4 允许和不允许的通道

该工具仅允许评估环境明确暴露的基准局部交互通道。允许的通道包括文件系统的配置局部、基准提供的Docker API和用于真空智能体设置的终端接口，以及用于检索基础方法的本地检索后端。禁止的通道包括公共网络搜索、外部检索语料库、未向方法公开的隐藏元数据，以及任何配置外部信息来源。对于交互智能体，多模态内容同样通过提供的接口进行介导，而不是默认提供。

# D.2 方法体系与工具配置

HippoCamp 在三种执行模式下评估方法，这三种模式在检索、工具使用和多模态感知的实现方式上有所不同：(i) 本地检索设置，方法直接在基准本地检索后端上操作；(ii) 真空Docker代理设置，终端式智能体通过受控的命令界面与特定配置的容器交互；(iii) 官方托管代理设置，适用于内部编排无法在本地准确重现的商业智能体产品。在这三种模式下，访问的信息仍然保持在配置本地。因此，各模式之间的比较旨在评估在匹配的配置本地信息访问下的基础任务表现，而不是声称在底层工具接口或执行能力上完全一致。

# D.2.1 本地检索设置

我们首先在本地检索设置中评估一组方法，其中系统直接在基准本地语料库上操作，而不是通过虚拟 Docker 终端环境。该组包括两个检索原生基线（标准 RAG 和自我 RAG）和三个搜索启用生成器（Search-R1 及两个 ReAct 变体）。尽管这些方法在交互风格上有显著差异，但它们共享相同的本地检索后端，并且不访问外部网络资源。该设置的目标是在保留共同语料库和搜索范围的情况下，比较方法在其原生检索或搜索循环下的表现。 标准 RAG（Lewis 等人，2020）。标准 RAG 实现为经典的检索-重排序-返回管道。给定一个查询，系统首先对基准本地向量库执行向量相似性检索，可选地对检索到的块进行重排序，并将前 $k$ 个结果返回给生成器。在我们的实现中，检索由初始检索文档的数量 $\left( \tt t o p \_ k \right)$、是否启用重排序以及最终重排序截止（rerank_top_k）参数化。该方法使用嵌入模型、向量存储和可选重排序器，但不进行迭代查询完善、工具使用或环境交互。因此，它在我们的比较中作为最简单的检索原生基线。 自我 RAG（Asai 等人，2024）。自我 RAG 在标准检索设置上扩展了一个内部反思阶段。系统首先从相同的基准本地向量库中检索候选块，然后使用生成器对每个检索项的相关性进行评分，过滤集合以仅保留高于相关性阈值的项目，并可选地重写查询并重复检索，直到找到相关证据为止。关键控制参数包括检索深度（top_k）、相关性阈值以及最大细化迭代次数。尽管存在自我反思循环，自我 RAG 仍然是一种检索原生方法，而不是完全交互的智能体，因为它没有通过外部环境接口进行操作，也不执行明确的文件级工具使用。 Search-R1（Jin 等人，2025）。Search-R1 被评估为具有内部搜索的端到端生成器。与 RAG 管道不同，它不依赖于暴露在评估环境中的单独检索提供者。相反，头部在检索过程中采用诸如 <think>、<search> 和 <answer> 的预定义结构。搜索查询在生成过程中动态发布，返回的结果被重新纳入模型上下文中以便后续推理。在我们的设置中，这些搜索是针对与其他原生方法相同的基准本地检索服务器执行的，确保搜索范围保持在本地并具有可比性。 ReAct（Yao 等人，2023）变体。ReAct 方法实现了经典的思维-行动-观察循环，其中模型明确推理所需的信息，发出搜索动作，从本地检索服务器接收观察，并迭代直到产生最终答案。两个 ReAct 变体共享相同的 ReAct 交互框架和相同的基准本地搜索后端；它们仅在底层生成模型上有所不同，即 Gemini-2.5-fash（Comanici 等人，2025）与 Qwen3-30B-A3B（Bai 等人，2025）。与标准 RAG 和自我 RAG 相比，ReAct 系统是搜索交互的，而不是纯粹的检索原生，但它们仍在 Docker 环境外进行评估，并且不依赖于外部网络访问。

# D.2.2 真空 Docker 智能体设置

接下来，我们在受控的真空Docker环境中评估三种终端风格的智能体：终端智能体（Qwen3-VL-8B-Instruct（Bai等，2025））、终端智能体（ChatGPT-5.2（OpenAI，2025b））和终端智能体（Gemini-2.5-fash（Comanici等，2025））。与原生检索设置不同，这些方法通过在特定配置文件的容器中暴露的统一终端风格接口与基准进行交互。这个设置旨在在受控条件下近似文件系统交互，同时保持不同智能体实现之间的可比性。所有三个智能体通过相同的容器驻留命令/API接口访问配置文件本地文件系统。该环境暴露了五个基准定义的原语。list_fles通过列出可访问数据根下的文件夹，提供目录级发现，支持基于模式的过滤，因此用于广泛搜索和导航。return_metadata在可用时返回结构化的文件属性，包括类型、模态、时间戳和位置相关字段，使元数据意识推理成为可能，而不揭示文件内容。return_txt返回文件的结构化文本表示（例如，提取的文本片段以及基本文件信息），成为文档和其他可解析格式的文本检查的主要接口。return_img将文件或选定页面呈现为图像，并返回生成的图像路径及其图像数据，支持页面级的视觉检查。最后，return返回实际文件路径和原始文件字节，使智能体能够在直接处理源文件时实现精准的访问。这些接口共同暴露了互补的抽象层级：发现（list_fles）、元数据检查（return_metadata）、文本级访问（return_txt）、渲染视觉访问（return_img）和原始字节访问（return）。该环境是受控的，而不是完全真实操作系统的等价物。智能体没有获得无限制的桌面交互；相反，所有文件访问均通过基准提供的命令和转换工具中介。因此，这种设置的评估价值在于，它不是忠实地再现一个消费者操作系统，而是提供了一个可重复的、配置文件隔离的、工具一致的终端基础用于智能体交互。由于所有终端智能体被限制为相同命令，因此直接操控它们的能力更好，能够在规划、搜索、处理返回内容和管理中介假设方面表现更佳，而不是依赖于工具可用性的差异。我们的设计选择是多模态访问驱动于终端结果，而非无条件可用。特别地，图像或原始文件渠道仅在智能体显式调用return_img或return_ori，并且命令成功时才可用。返回的数据随后被转换为相应智能体后端使用的特定多模态输入格式。因此，在该环境中的多模态感知并非无代价：它必须通过故意使用工具来触发，就像文件发现和元数据检查必须通过显式命令触发一样。

# D.2.3 Docker 工具及多模态返回路径

该真空 Docker 环境建立在一个基于 Ubuntu 的镜像上，打包了与受控智能体交互所需的所有配置本地资源，包括基准数据、元数据、金标准文本表示、辅助工具以及轻量级的 WebUI/API 层。除了标准的文件服务外，该容器还包括一小组转换工具，使异构个人文件能够通过统一接口进行访问，包括用于办公文档的 LibreOffice、用于 PDF 渲染的 Poppler/pdf2image、对结构化数据库文件的 SQLite 支持，以及专用的元数据/图像服务端点。其目标并不是模拟完整的桌面操作系统，而是为基于终端的智能体提供一个可重复、工具一致的基础环境。在该环境中，所有终端智能体通过相同的命令界面进行交互：istfiles、return_txt、return_img、return_ori 和 return_metadata。这些命令定义了对底层文件系统的共同抽象，从发现和元数据检查到文本提取、渲染访问以及原始响应。由于命令清单涵盖所有终端智能体，因此在接口层面强制执行平等性：每个方法都获得对相同基准本地功能的访问，而性能差异源于模型如何有效规划和利用这些工具，而不是工具的可用性本身。公平性的重要方面是默认情况下不暴露多模态输入。相反，多模态访问是由终端结果驱动的：智能体仅在明确调用 return_img 或 return_ori 并且命令返回成功 $=$ true 后才进入多模态通道。因此，视觉或源级内容必须通过有意的工具使用进行请求，而不是被动地注入到模型上下文中。这使得多模态感知成为智能体解决问题的负担的一部分，伴随于文件搜索和证据定位。返回的多模态有效载荷随后通过模型特定的传输路径进行消费。对于与 ChatGPT 兼容的终端智能体，渲染的图像作为图像块附加或作为数据 URL 有效载荷嵌入。对于基于 Gemini 的终端智能体，相同的输出在下一个模型调用之前转换为本地多模态部分（例如，上传的图像/文件内容）。在这两种情况下，基准级别的命令语义保持一致；不同之处仅在于返回的工件在模型输入通道中的后端特定序列化。这种分离是重要的：环境平衡了可以请求的信息，而模型后端仅决定如何摄取这些返回的信息。

# D.2.4 官方托管智能体设置

除了本地检索和 vacuum-Docker 设置，我们还评估了 ChatGPT 智能体模式（OpenAI 等，2024；OpenAI，2025a），其官方托管配置由 OpenAI 提供。该设置对应于商业产品的部署，而不是本地控制的研究环境。因此，该方法不通过我们的 vacuum Docker 接口运行，也不受限于终端智能体相同的命令/API 接口。我们明确将此配置区分为原生托管智能体模式。这个区分很重要，因为其内部工具、编排策略和多模态处理由产品平台管理，而不是由我们的评估框架管理，因此无法完全与基于 Docker 的智能体标准化。不过，我们将其纳入比较中，因为它代表了实际部署的智能体系统的一个强大且现实相关的参考点。换句话说，尽管它与 vacuum-Docker 智能体并非严格并行工具，但它为在官方使用条件下，先进的托管智能体在 HippoCamp 上所能达到的水平提供了一个重要的上限式比较。

# D.3 预算、重试与随机性

由于HippoCamp包含长期、多模态任务，评估结果对资源分配敏感。因此，我们明确资源预算，并将其解释为评估协议的一部分，而不是隐藏的实现细节。我们不强制在所有系统中采用单一的全球匹配预算；相反，每种方法在其本机实现或服务环境中，都会在预先指定的、适合该方法的预算下进行评估。相关的约束包括搜索迭代次数、检索证据量、生成长度、实际运行时间以及特定方法的停止规则。

# D.3.1 资源预算

我们将预算体制分为两大类。

检索原生方法。对于检索原生系统，包括标准 RAG 和自我 RAG，主要的预算维度是检索深度、重排序/反思深度和生成长度。标准 RAG 使用一个有界的检索预算，最初检索的块数有限，具有可选的重排序阶段，以及最终的重排序截止点；它不执行迭代搜索或环境交互，因此其有效预算主要由检索的前 $k$ 条、重排的前 $k$ 条和答案模型的最大生成长度决定。自我 RAG 在相同的语料库本地检索设置下工作，但为自我反思分配了额外的预算，包括有界的相关性评分、过滤、可选的查询重写和有限数量的改进迭代。在这两种情况下，只要检索或生成管道返回没有进一步有效证据，就启用早期停止，并且实际运行时间受限于底层检索/生成调用的完成，而不是外部交互循环。交互智能体。对于交互方法，包括 Search-R1、ReAct 变体、终端智能体和 ChatGPT 智能体模式，相关的预算维度更为异构。Search-R1 和基于 ReAct 的系统受限于最大搜索或推理轮次、每次搜索调用返回的检索结果数量、每轮的最大生成标记以及任何内置的停止条件（例如，当生成最终答案行为时终止）。在真空 Docker 环境中的终端智能体还受到终端交互预算的限制，包括工具使用迭代次数和完成查询所需的实际时间。ChatGPT 智能体模式是在托管产品设置的实际限制下进行评估的，这些限制包括平台侧对交互长度、生成预算和运行时的约束。在所有交互方法中，只要方法的本地控制循环以最终答案终止或在剩余预算内无法进行进一步的有效交互，就会使用早期停止。

# D.3.2 重试与失败处理

由于多个评估系统是交互式和工具使用的，故障不仅可能源于推理错误，还可能由于格式错误的动作、不完整的输出或环境执行问题。因此，我们明确指定了重试处理。一般而言，我们不会重试成功但不正确的答案：重试仅保留在未能产生有效可评估输出的情况下。具体而言，重试触发条件包括 (i) 无法解析为最终答案的格式错误输出，(ii) 空答案，(iii) 工具故障导致无法访问请求的基准本地资源，以及 (iv) 严格的超时。当发生此类故障时，系统将在相同的查询下在相同的环境和预算约束下重新执行；成功但不正确的答案将被视为失败，而不是重试。然后根据该框架下每种评估方法的预先指定执行策略生成报告结果。这个区分对于不稳定的交互式智能体尤为重要，因为执行故障可能会实质性影响最终测量结果。

# D.3.3 随机性与确定性

我们在基础方法允许的情况下控制随机性，但并非所有评估的系统都能实现严格的确定性。对于检索原生方法，随机性受到限制，并且可以通过固定种子和确定性的检索设置在很大程度上进行控制，变异主要来自于启用时的随机生成组件。对于交互式智能体方法，额外的非确定性通过搜索顺序、多轮解码、工具使用分支和后端特定采样行为引入。因此，我们报告每种方法使用的采样配置，包括适用时的生成温度。托管商业API引入了进一步的可变性源，因为其内部服务堆栈和解码行为并未完全暴露给用户。特别是，当提示和可见设置固定时，托管的智能体模式可能仍然保持非确定性。因此，特别对于长期交互方法，运行时的方差应当得到预期。当因成本或平台限制而无法进行所有方法的重复运行时，我们会明确指出，并将结果解释为在相应执行模式下的点估计，而不是完全稳定的平均值。这种处理旨在使残余方差可见，而不是通过对确定性的不完全声明隐性隐藏它。

# D.4 衡量标准与评估稳健性

HippoCamp 的设计不仅评估方法是否达到正确的最终答案，还检验其在现实个人情境下获取必要证据和运用适当能力的情况。因此，我们报告了三个互补的级别指标：答案质量、证据检索质量和能力表现。这种分解与基准测试的核心目标相一致，即诊断搜索、感知和推理中的失败，而不是将行为归纳为单一分数。

# D.4.1 回答质量

我们使用 LLM 作为评判者的协议来评估答案质量，该协议返回二元正确性决策和分级语义得分。评判者对所有示例进行初步评估，同时对抽样案例应用分层手动审计以验证判断质量。对于每个查询，评判者接收问题、模型预测和真实答案，并输出：（i）一个二元标签 pred $\in \{ \tt y e s , n o \}$，指示预测是否在语义上可接受，以及（ii）一个整数得分 $s _ { i } \in [ 0 , 5 ]$，用于在粗略的语义尺度上衡量答案质量。准确率（Acc）。我们将准确率（同样为通过率）报告为标记为是的判断响应所占的比例：

$$
\mathrm { A c c } = { \frac { \# ( { \mathrm { p r e d } } = \mathbf { y } \mathbf { e } \mathbf { s } ) } { N } } ,
$$

其中 $N$ 是已评估实例的数量。在汇总结果文件中，该数量也被报告为通过率（Pass Rate）。它衡量了在评审者接受标准下，方法是否产生了语义正确的答案。平均评审得分。对于一组 $N$ 个实例，我们计算平均评审得分为

$$
\bar { s } = \frac { 1 } { N } \sum _ { i = 1 } ^ { N } s _ { i } , \qquad s _ { i } \in [ 0 , 5 ] .
$$

在每个领域总结的 CSV 导出中，我们还报告了重新缩放的平均得分 (/10)：

$$
{ \mathrm { A v g S c o r e } } _ { / 1 0 } = 2 \cdot { \bar { s } } ,
$$

使得报告的值位于 010 等级上。该分级分数与二元准确率互为补充：两种方法可能实现类似的通过率，但在答案的完整性、精确性或支持程度上存在实质性差异。基准级别的聚合。对于方法 $m$，在所有评估实例上计算全球基准平均值，方法是对所有经过评判的实例进行样本加权均值计算：

$$
\mathrm { B e n c h m a r k A v g } _ { m } = \frac { \sum _ { d } \sum _ { i = 1 } ^ { N _ { m , d } } s _ { m , d , i } } { \sum _ { d } N _ { m , d } } ,
$$

其中 $d$ 表示配置，$N_{m,d}$ 是方法 $m$ 在配置 $d$ 上被评判实例的数量。我们还报告相应的人群标准差，其中 $\mu_{m} = \mathrm{Benchmark \, Avg}_{m}$，且 $\begin{array}{r} N_{m} = \sum_{d} N_{m,d} \end{array}$。

$$
\mathrm { B e n c h m a r k S t d } _ { m } = \sqrt { \frac { 1 } { N _ { m } } \sum _ { j = 1 } ^ { N _ { m } } { \left( s _ { m , j } - \mu _ { m } \right) ^ { 2 } } } ,
$$

这些答案级别的指标直接反映了基准测试的主要最终目标：一个系统是否能够生成基于用户本地文件系统的正确且有用的响应。

# D.4.2 证据检索指标

仅仅回答的正确性对于 HippoCamp 来说是不够的，因为一种方法可能会给出一个看似合理的答案，但未能检索到或依据必要的证据。因此，我们根据注释的最小支持文件集来评估检索质量。令 $G _ { i }$ 表示实例 $i$ 的真实标注文件集，$\hat { R } _ { i }$ 表示该方法检索或引用的文件集。我们计算文件级的精确度、召回率和 F1 值，如下所示：

$$
P _ { i } = { \frac { \left| G _ { i } \cap { \hat { R } } _ { i } \right| } { \left| { \hat { R } } _ { i } \right| } } , \qquad \mathrm { R e c } _ { i } = { \frac { \left| G _ { i } \cap { \hat { R } } _ { i } \right| } { \left| G _ { i } \right| } } , \qquad F 1 _ { i } = { \frac { 2 P _ { i } \operatorname { R e c } _ { i } } { P _ { i } + \operatorname { R e c } _ { i } } } .
$$

报告的文件精度、文件召回率和文件 $F 1$ 通过对评估分割中的这些实例级别值进行平均得到。文件命中率。我们还报告文件命中率，在我们的实现中对应于平均文件级别召回率：

$$
\mathrm { H i t R a t e } = \frac { 1 } { N } \sum _ { i = 1 } ^ { N } \mathrm { R e c } _ { i } .
$$

这个数量反映了该方法覆盖所需支持证据的程度，即使它也会检索到虚假文件。用于分析任务的解释。对于分析任务，支持证据通常由分散在多个文件和时间点的弱信号组成。因此，文件级 F1 总是相对于标注的最小支持文件集进行计算，而不是针对所有潜在相关的文件进行计算。这个区别很重要：检索指标旨在衡量所需证据的覆盖性，而不是方法是否重现了人类推理路径的确切过程，或检索到了每个可能对同一推理产生贡献的文件。这些检索指标对于 HippoCamp 的诊断角色至关重要：它们将搜索和证据覆盖的失败与已检索内容的推理失败分开。

# D.4.3 能力指标

为了进一步本地化故障模式，我们利用基准测试中人工标注的 agent_cap 标签报告按能力分类的性能。每个实例可能对应于一个或多个能力箱，这些箱子对应于搜索、证据感知和推理。对于每个能力类别，我们首先计算其组成子类别的性能，然后汇总这些子类别的统计数据。按能力分类的准确率。对于给定的子类别 $c$ ，令 ${ \mathrm { y e s } } _ { c }$ 表示判断为正确响应的数量，count $c$ 表示标注为该子类别的实例数量。子类别准确率为

$$
\operatorname { a c c } _ { c } = { \frac { \operatorname { y e s } _ { c } } { \operatorname { c o u n t } _ { c } } } .
$$

报告的能力家族准确性（例如，搜索/证据/推理准确性）是相应子类别的无权算术平均值：

$$
{ \mathrm { A c c } } _ { \mathrm { f a m i l y } } = { \frac { 1 } { | C | } } \sum _ { c \in C } \operatorname { a c c } _ { c } ,
$$

其中 $C$ 是该家族中子类别的集合。能力方面的 F1。类似地，如果 $\mathrm{f}1_{c}$ 表示与子类别 $c$ 相关的平均文件级 F1 或证据级 F1，则报告的家族级 F1 为

$$
{ \mathrm { F } } 1 _ { { \mathrm { f a m i l y } } } = { \frac { 1 } { | C | } } \sum _ { c \in C } { { \mathrm { f } } 1 _ { c } } .
$$

我们使用无权重而非样本加权的平均计算，以避免广泛的能力类别被其最常见的子类别所主导。这使得生成的指标更适合于对异质推理技能进行诊断比较。延迟。最后，为了完整性，我们报告平均延迟，作为有效每个实例运行时间的均值：

$$
\mathrm { A v g L a t e n c y } = \frac { 1 } { k } \sum _ { i = 1 } ^ { k } t _ { i } , \qquad t _ { i } > 0 ,
$$

只有记录了正运行时间的实例被纳入考虑。延迟并不被视为主要质量指标，但它对于理解各方法家族之间能力与效率的实际权衡是有用的。总体而言，这一指标套件反映了HippoCamp的基准理念：一个优良的方法不仅应该生成正确答案，还应该通过检索正确证据并运用适当的搜索、感知和推理能力，以一种可靠且可审计的方式来实现。

# D.4.4 LLM作为裁判的鲁棒性

由于HippoCamp的大部分内容需要开放式、基于证据的回答，而不是精确字符串匹配，因此我们采用了基于大型语言模型（LLM）进行语义评估的协议。我们将评审输出视为一种受控的语义信号，而非独立的神谕，因此明确了评审设置、提示约束和审计程序。 评审设置：对于每个被评估的实例，评审者接收三个输入：问题、真实答案和模型预测。评审者被指示评估语义匹配，而不是词汇重叠，允许释义和不冲突的扩展，同时惩罚关键信息的遗漏。输出包括一个二元决定 $\mathsf { d } \in \{ \mathsf { y e s } , \mathsf { n o } \}$ 以及一个整数评分 $s \in [ 0 , 5 ]$。这提供了一个严格的正确性信号和一个分级的质量信号，这些信号在我们的答案级别评估中使用。 为了减少无关的变化，所有输入在评审之前都被规范化为固定的提示格式。评审者只比较问题、真实答案和候选答案；它无法访问诸如能力标签、推理轨迹或真实证据集等隐性基准注释。此外，候选响应在没有明确系统身份的情况下呈现，以便提示不会直接暴露出哪个模型生成了该答案。指令故意受到限制：它优先考虑语义等价，明确容许释义和同义词，并要求最小的JSON输出，仅包含预测和评分，使评审过程易于解析、审计和重现。以下展示了一个代表性的评审指令。

该研究在答案层面的轮廓方面对不同方法进行了比较。例如，终端智能体（GPT-5.2）在轮廓任务上的准确率从30.0提升到45.7，而ChatGPT智能体模式则从48.3提高到56.8。这一模式与预期任务设计是一致的。事实保留问题通常更依赖于恢复明确的用户本地事实，而轮廓任务则需要从薄弱、分散且通常时间上分离的信号中进行抽象。值得注意的是，这一答案级别的差距比检索质量的相应差距要大。对于几种方法，轮廓的File F1并没有显著低于事实保留的File F1，且在某些情况下相对于该方法家族仍然保持可比性。因此，轮廓准确率的大幅下降指向了一种推理瓶颈，而不单纯是检索瓶颈：即使候选证据部分可用，稳定的用户特征、偏好和常规依然比重述本地化事实要复杂得多。第四，RAG管道展示了特征覆盖和特定性之间的权衡。标准RAG和自我RAG实现了相对较高的总体File Recall（69.3和61.3），但它们的整体File Precision仍然有限（21.7和24.5），且总体准确率依然适中（27.5和23.8）。这一模式表明，这些方法通常会检索到与注释支持集重叠的广泛证据池，但在孤立出精确扎根回答所需的最小证据方面存在困难。该效果在事实保留上尤为明显，尽管两种方法恢复了大量相关文件，但在判断上的正确性表现仍然大幅低于期望。这一结果与HippoCamp的更广泛设计动机是一致的：基准成功不仅需要粗略的检索覆盖，还需要严谨的证据选择和对异质个人文件的可靠推理。最后，存在明显的效率与能力之间的权衡。标准RAG是最快的方法，平均延迟在所有三种聚合中最低，而ReAct（Gemini-2.5-fash）和Search-R1与完全自主智能体相比仍然相对高效。相反，最强的答案级方法，即ChatGPT智能体模式，其延迟远高于其他方法。同样的趋势在其他具有更强终端任务性能的智能系统中也有所体现，尽管不如此极端。这表明，当前在扎根的多模态文件推理中的收益部分是通过更长的交互周期、更多的迭代工具使用或更重的跨文件处理来实现的，而不仅仅是通过更高效的推理。总体而言，表8中的结果进一步强化了HippoCamp的中心论点：个性化文件系统的问答不能仅通过最终准确率来充分描述。那些在检索上看似具有竞争力的方法在证据解释和综合时可能仍然失败，而那些实现更强最终正确性的方法往往伴随着显著的计算开销。因此，该基准暴露了证据覆盖、扎根推理和效率之间的三方紧张关系，而这在单一指标评估中往往被掩盖。
